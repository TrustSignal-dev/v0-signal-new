!eyebrow Private Partner Concept

!privateNote This page is private partner walkthrough material for Vanta and TrustSignal. It is an integration concept only, not a public product claim, not an announced native integration, and not a statement of GA partner functionality.

!summary This concept shows how Vanta can remain the evidence workflow and system of record while TrustSignal performs an integrity check, returns integrity verification signals, and issues a signed verification receipt for later comparison against the previously checked artifact state.

!context Compliance artifacts can be collected correctly and still become harder to trust later if the file changes, the export drifts, or the evidence is separated from the context that shows what was previously checked.

!integration Vanta remains the workflow and system of record. TrustSignal receives the artifact or artifact reference, performs an integrity check, returns integrity verification signals, and issues a signed verification receipt that can be associated with the evidence record for later review.

!whyPartner This gives Vanta teams a durable way to show what artifact state was previously checked and whether the evidence presented later still matches that state before a reviewer relies on it.

!talkTrack Start with an evidence record already living in Vanta. Send the artifact hash and record reference to TrustSignal. Read back the integrity verification signals and signed verification receipt. Associate the receipt with the evidence record. Later, compare the artifact presented at review time against the previously checked state and surface any mismatch.

!productionNote In a production deployment, the integrity check request should be authenticated, the receipt stored alongside the evidence record, and later comparison invoked before a reviewer relies on a previously checked artifact.

This example starts with a compliance artifact already tracked in Vanta. Vanta keeps the workflow state, owner, and control context.

!narration In the walkthrough, the evidence already lives in Vanta before TrustSignal is introduced.

## !!steps Integrity check request

The TrustSignal request carries the evidence reference, artifact hash, and workflow context needed to bind the integrity check to the record that Vanta already manages.

!narration This request shows additive workflow infrastructure rather than a system replacement.

!takeaway The request creates a stable bridge between Vanta's evidence record and TrustSignal's signed verification outputs.

```json !code integrity-check-request.json
{
  "subject": {
    "system": "vanta",
    "record_id": "ve_01JQ9ZQTRP4H4BDM7Q0M6F0NX5",
    "control_id": "CC-3.2",
    "artifact_reference": "s3://vanta-evidence/identity/sso-config-2026-03-13.json",
```

## !!steps Integrity verification signals

TrustSignal returns integrity verification signals in workflow-friendly language and avoids internal implementation detail.

!narration This is the operational outcome: the supplied artifact state matched and a signed verification receipt was issued.

!takeaway The result is usable in the workflow immediately and points to a signed receipt that can be retained for later comparison.

## !!steps Signed verification receipt

!narration The receipt is the durable integrity record that stays associated with the evidence workflow.

## !!steps Later comparison and mismatch detection

Later, a reviewer can compare the evidence presented at review time against the stored receipt context. If the artifact has changed, TrustSignal can return a mismatch instead of silently reusing the earlier result.

!narration This is where later comparison becomes valuable: it can surface drift before a reviewer relies on the evidence record.

!takeaway Drift detection matters because it surfaces changes at review time without asking Vanta to stop being the system of record.

```json !code later-comparison-mismatch.json
{
  "receipt_id": "rcpt_01JQ9ZZ3M57Y8R4QX3P8V9S6KD",
  "integrity_verified": false,
  "reason": "artifact_mismatch",
  "stored_artifact_hash": "sha256:3f1b483f00cd78b4d8f54e5a7fd9e55c86412f3275ef7d9cd3cc15364869c0be",
  "presented_artifact_hash": "sha256:be31af6ac4d0e7b22960557741331d6179a53f7d95a519772e487f1398cb4cf7",
  "reviewed_at": "2026-04-09T14:40:31.000Z",
  "action": "request_fresh_evidence"
}
```
