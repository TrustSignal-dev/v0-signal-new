!eyebrow Private Partner Concept

!privateNote This page is private partner walkthrough material for ICE Mortgage Technology and TrustSignal. It is an integration concept only, not a public product claim, not an announced native integration, and not a statement of GA partner functionality.

!summary This concept shows how Encompass can remain the loan file workflow and system of record while TrustSignal performs an integrity check at each material loan event, returns integrity verification signals, and issues a signed verification receipt that can be associated with the loan file for later comparison at post-close audit or repurchase review.

!context Mortgage loan files are touched by many parties between application and post-close. The artifact collected at intake may be reviewed again years later during a post-close audit or a repurchase demand. The question that decides outcomes at that point is whether the artifact presented at review still matches the artifact that was checked when the workflow step happened.

!integration Encompass remains the workflow and system of record for the loan file. TrustSignal receives the artifact or artifact reference at material loan events, performs an integrity check, returns integrity verification signals, and issues a signed verification receipt that can be associated with the loan record for later review. TrustSignal does not store the loan file, does not replicate Encompass workflow state, and does not act as a system of record for the loan.

!whyPartner This gives mortgage operations and post-close teams a durable way to show what artifact state was previously checked at each material loan event and whether the evidence presented later still matches that state before relying on it for an audit response or a repurchase rebuttal.

!talkTrack Start with a loan file event already happening in Encompass, such as application intake, an income document delivery, an appraisal receipt, or a closing package assembly. Send the artifact hash and loan reference to TrustSignal. Read back the integrity verification signals and signed verification receipt. Associate the receipt with the loan record. Later, at post-close audit or in response to a repurchase demand, compare the artifact presented at review against the previously checked state and surface any drift before relying on the file.

!productionNote In a production deployment, the integrity check request should be authenticated, the receipt stored alongside the loan record, and later comparison invoked before relying on a previously checked artifact for an audit response or a repurchase rebuttal.

!takeaway Encompass remains the system of record. TrustSignal receives stable artifact material or a stable reference plus loan context.

## !!steps Integrity check request

The TrustSignal request carries the loan reference, document type, artifact hash, and event context needed to bind the integrity check to the loan record that Encompass already manages.

!narration This is additive workflow infrastructure. The request is about the loan file event and its checked state, not about replacing Encompass workflow logic.

!takeaway The request creates a stable bridge between the Encompass loan record and TrustSignal's signed verification outputs.

```json !code integrity-check-request.json

  "subject": {
```

## !!steps Integrity verification signals

TrustSignal returns integrity verification signals in workflow-friendly language and avoids internal implementation detail.

!narration This is the operational outcome: the artifact matched the previously supplied state and a signed verification receipt was issued.

!takeaway The result is usable in the loan workflow immediately and points to a signed receipt that can be retained for later comparison at post-close audit or repurchase review.

## !!steps Signed verification receipt

!narration The receipt is the durable integrity record. It travels with the loan record instead of replacing any Encompass workflow data.

!takeaway The receipt binds the loan reference, the document state, the integrity verification outcome, and durable receipt metadata into a record that supports later comparison at audit time.

## !!steps Later comparison at post-close audit or repurchase review

Later, a reviewer can compare the document presented during a post-close audit or in response to a repurchase demand against the stored receipt context. If the artifact has changed since the event was checked, TrustSignal can return a mismatch so the team can request fresh evidence before relying on the file.

!narration This is where the value lands: at post-close audit or repurchase review, the receipt supports comparison against the previously checked state and can surface drift before the team responds.

!takeaway Drift detection matters because it surfaces changes at review time without asking Encompass to stop being the system of record for the loan file.

```json !code later-comparison-mismatch.json
{
  "receipt_id": "rcpt_01JQ9X07S9PKS0YQG7B8W6X1CN",
  "integrity_verified": false,
  "reason": "artifact_mismatch",
  "stored_artifact_hash": "sha256:0f3d43a0a6c70f3b4bdb5c8a8e3d8fd4c3a2d4f2836e1f8431d0a0a7984e4c8a",
  "presented_artifact_hash": "sha256:7d02a6e4f5cbfd616cb50cbeeaec4d37c78f5fc0bd385b0ecff7af9afbd4ab11",
  "reviewed_at": "2026-04-07T09:12:44.000Z",
  "action": "request_fresh_evidence"
}
```
